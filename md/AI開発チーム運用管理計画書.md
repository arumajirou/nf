# AI開発チーム運用管理計画書

## エグゼクティブサマリー

本文書は、ChatGPT-5 Thinking（プロジェクトマネージャー）、人間の管理者（意思決定者）、Claude Code（実装担当）の3者体制によるAI駆動型開発プロジェクトの運用管理方法を定義します。

**最重要課題**: 人間の気分や稼働状況に依存しない、自動化されたPDCAサイクルの構築

---

## 1. Claude Codeの特性分析（公式ドキュメント調査結果）

### 1.1 Claude Codeとは

Claude Codeは、Anthropic社が開発したエージェント型コーディングツールで、以下の特徴を持ちます:

**主要機能:**
- コードベース全体の理解と解析
- ファイルの直接編集とコマンド実行
- Git操作（commit, PR作成）の自動化
- 自然言語での指示受付
- リアルタイム進捗トラッキング

**利用可能なインターフェース:**
- ターミナル CLI（`npm install -g @anthropic-ai/claude-code`）
- VS Code拡張機能
- Webインターフェース（claude.ai/code）
- GitHub統合（@claude タグ）

**並列実行機能:**
- Git worktreesを使用した複数インスタンス同時実行
- 独立したタスクの並列処理
- 各インスタンスは隔離された作業ディレクトリを保持

### 1.2 推奨ワークフロー（Anthropic公式ベストプラクティス）

```
ステップ1: リサーチ
→ 問題の理解、既存コードの調査、関連ドキュメントの確認

ステップ2: プランニング
→ 解決策の設計、タスクの分解、実装計画の作成

ステップ3: 実装
→ コードの記述、段階的な検証、合理性チェック

ステップ4: コミット＆PR
→ 変更の記録、PR作成、README/CHANGELOG更新
```

**重要な洞察:**
- ステップ1-2を省略すると品質が低下する
- 明示的な検証ステップを含めることで精度向上
- テスト駆動開発との相性が良い

### 1.3 高度な使用パターン

**複数エージェント協働:**
- テスト作成専用エージェントとコード実装エージェントの分離
- スクラッチパッドを使った エージェント間通信
- 独立タスクの並列処理による効率化

**自動化可能なタスク:**
- lint エラー修正
- マージコンフリクト解決
- リリースノート作成
- PR コメントへの対応
- ユニット/統合/E2E テストの実行と修正

## 2. 現状の課題分析

### 2.1 構造的問題

**依存関係の問題:**
```
人間の管理者 ─┐
              ├→ 進捗確認 ─→ 指示出し ─→ ChatGPT-5 ─→ Claude Code
              │
              └─ 進捗が滞る（人間が動かない場合）
```

**ボトルネック:**
1. 人間の管理者が現状把握しないと進まない
2. 人間の気分・体調・時間に依存
3. 非エンジニアが技術的判断を求められる
4. 日次PDCA サイクルが不安定

### 2.2 根本原因

- **情報の非対称性**: 非エンジニアが技術詳細を理解困難
- **プロセスの非自動化**: 手動トリガーに依存
- **役割の曖昧性**: 誰が何を判断すべきか不明確
- **可視化の欠如**: 進捗状況のリアルタイム把握が困難

## 3. 解決策: 自動化されたPDCAシステム

### 3.1 システムアーキテクチャ

```
┌─────────────────────────────────────────────────────┐
│         自動化レイヤー（専用GPTs）                     │
│  - 日次進捗レポート自動生成                            │
│  - 優先度付きタスクキュー管理                          │
│  - 異常検知と人間へのエスカレーション                   │
└─────────────────────────────────────────────────────┘
              ↓ 自動 ↑ 必要時のみ人間介入
┌─────────────────────────────────────────────────────┐
│      ChatGPT-5 Thinking（PM/リードエンジニア）         │
│  - 技術的判断                                         │
│  - タスク分解とスケジューリング                         │
│  - Claude Code への詳細指示生成                       │
└─────────────────────────────────────────────────────┘
              ↓ 指示 ↑ 実行結果
┌─────────────────────────────────────────────────────┐
│        複数Claude Code インスタンス                    │
│  Instance 1: 機能A開発                               │
│  Instance 2: バグ修正                                │
│  Instance 3: テスト作成                              │
└─────────────────────────────────────────────────────┘
```

### 3.2 自動化の鍵: 3層の意思決定階層

#### 層1: 完全自動実行（人間不要）
- コードの lint 修正
- ユニットテストの作成と実行
- ドキュメント更新
- 既存バグの修正（定義済み）
- 定期的なリファクタリング

#### 層2: 事前承認制（週次/月次レビュー）
- 新機能の追加（要件定義済み）
- アーキテクチャ変更（小規模）
- 依存関係の更新
- パフォーマンス最適化

#### 層3: 人間判断必須（即時エスカレーション）
- ビジネス要件の変更
- セキュリティリスクの検出
- 予算超過の懸念
- アーキテクチャの大幅変更

### 3.3 日次自動PDCAサイクル

#### 時刻 00:00 - 前日サマリー生成
```yaml
トリガー: 定時実行（GitHub Actions / cron）
実行者: 専用GPTs
処理:
  1. 全Claude Code インスタンスの作業ログ取得
  2. Git コミット履歴分析
  3. テスト結果集計
  4. 進捗率計算
  5. 人間向けサマリー生成（非技術用語）
出力: 日次レポート（Slack/Email/Notion）
```

#### 時刻 01:00 - 次日タスク計画
```yaml
実行者: ChatGPT-5 Thinking
入力: 
  - 前日サマリー
  - 未完了タスクリスト
  - 優先度マトリクス（事前定義）
処理:
  1. 前日の成果評価
  2. 新たな問題の識別
  3. 次日タスクの優先順位付け
  4. Claude Code への指示生成
  5. リソース配分（複数インスタンス）
出力: 
  - 実行計画書
  - Claude Code 用プロンプト集
  - 人間への報告（承認不要な場合は情報共有のみ）
```

#### 時刻 09:00 - 人間への朝報
```yaml
配信方法: Slack/Email/専用ダッシュボード
内容:
  - 昨日の成果（ビジネス視点）
  - 今日の実行計画（承認不要項目）
  - 要判断事項（ある場合のみ）
  - 推定完了時刻
形式: 
  - 非技術者向け平易な言葉
  - 視覚的進捗バー
  - ワンクリック承認/却下ボタン
```

#### 時刻 10:00 - Claude Code 実行開始
```yaml
実行方法: GitHub Actions または専用スクリプト
並列実行:
  - Instance 1: 優先度A タスク
  - Instance 2: 優先度B タスク
  - Instance 3: テスト・検証
監視:
  - リアルタイム進捗トラッキング
  - エラー検知と自動リトライ
  - 異常時は ChatGPT-5 に報告
```

#### 時刻 18:00 - 夕方進捗報告
```yaml
配信: 人間の管理者へ
内容:
  - 今日の達成事項
  - 残タスク
  - 明日への引き継ぎ事項
  - 緊急対応が必要な問題（あれば）
```

## 4. 専用GPTs設計

### 4.1 GPTs の役割定義

#### GPT A: 「プロジェクトオーケストレーター」
**目的**: 人間とAI開発チームの橋渡し

**主要機能:**
1. 日次/週次/月次レポート自動生成
2. 人間からの曖昧な指示を技術仕様に変換
3. 優先度判断とタスクキュー管理
4. エスカレーション判定
5. 非技術者向けダッシュボード更新

#### GPT B: 「Claude Code マネージャー」
**目的**: Claude Code への最適な指示生成

**主要機能:**
1. タスクを Claude Code 用プロンプトに変換
2. 公式ベストプラクティスに基づく指示最適化
3. 複数インスタンスのタスク割り当て
4. 実行結果の評価とフィードバック
5. エラー発生時の代替案提示

#### GPT C: 「品質保証エージェント」
**目的**: 自動レビューと品質担保

**主要機能:**
1. コードレビュー自動実行
2. テストカバレッジ分析
3. セキュリティ脆弱性スキャン
4. パフォーマンス回帰検出
5. PR マージ可否判定

## 7. リスクと対策

### リスク1: AIの誤判断
**対策:**
- 重要変更は必ず人間レビュー
- ロールバック機能の整備
- 段階的な権限付与

### リスク2: コスト超過
**対策:**
- API 使用量の日次モニタリング
- 予算アラート設定
- 優先度ベースのリソース配分

### リスク3: セキュリティ懸念
**対策:**
- 機密情報の環境変数化
- アクセス権限の最小化
- 定期的なセキュリティ監査

### リスク4: システム依存度の増大
**対策:**
- 手動実行手順の文書化
- 定期的なフォールバック訓練
- 重要知識のチーム共有

## 付録: 重要な設計原則

### A. 人間をボトルネックにしない
- デフォルトは「自動実行」
- 人間の判断は「例外時のみ」
- 情報は「プッシュ型」で提供

### B. 透明性の確保
- すべての判断根拠を記録
- いつでも履歴を追跡可能
- 人間がオーバーライド可能

### C. 段階的な自動化
- 最初は人間が監督
- 徐々に信頼を構築
- 最終的に完全自動化

### D. 失敗から学ぶ
- エラーを記録し分析
- プロンプトを継続的改善
- ベストプラクティスを蓄積
